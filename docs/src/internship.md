## 第一章 基本实习情况

### 1.1 实习项目与实习单位概况

#### 1.1.1 实习项目简介

本人参加的实习项目是公司的主要产品线之一——EduSoho 在线网校系统企培版的新模块-知识库的开发。

> 母产品简介：

Edusoho 企培版——为企业快速发展赋能。官网介绍：<https://www.edusoho.com/>（主要）、<https://ct.edusoho.com/>（企培版）。

> 本产品背景：

- Situation：

企业中随着业务的发展，会产生大量的知识、经验文档等希望可以存放并供员工使用和学习，员工需要可以即查即用。尤其是涉及大量的业务知识的文档，将这些业务知识快速的同步给一线员工是这些企业的刚性需求，帮助员工解决工作中遇到的业务问题。

- Problems：

企业的知识、经验文档没有统一的有序的存放持续的沉淀，员工需要使用内容的时候，很难找到，散落在各处。内容更新不及时，版本混乱。

EduSoho 企培版中目前针对企业内部的知识以及经验的内容主要以线上课程的形似来沉淀和呈现，更适合用于体系化的内容学习，在知识库场景下不能很好满足员工即查即看即用的需求。另外从课程的内容呈现形式，课程的内容学习列表，不能满足知识管理和存放以及呈现上的需求。

#### 1.1.2 实习单位与实习指导老师

本人实习的单位是杭州阔知网络科技有限公司，部门为研发中心-主程序团队。阔知是一家一站式在线教育解决方案服务商，致力于为企业、个人和教育机构提供包括平台、运营、内容和营销在内的在线教育整体解决方案。阔知旗下拥有 EduSoho 网校、EduSoho 教育云、EduSoho 企培、EduSoho 智慧课堂等产品，截止 2019 年 1 月，已服务国内外近 10 万家企业和教育机构，200 多家高等院校，覆盖至美、加、日、英、德等多个发达国家。

实习的校内指导老师为汤颖。企业指导老师为洪永春，为该公司的前端开发工程师。

#### 2.2.3 产品技术方案

> 前后端分离

知识库技术实现上采用前后端分离的架构，前后端分离已成为互联网项目开发的业界标准使用方式。
在前后端分离架构中，后端只需要负责按照约定的数据格式向前端提供可调用的 API 服务即可，前端工程师只需要编写 HTML、js、CSS 等前端资源，然后通 过 HTTP 请求调用后端提供的服务即可。前后端之间通过 HTTP 请求进行交互，前端获取到数据后，进行页面的组装和渲染，最终返回给浏览器。
前后端代码库同样也分离，前端代码中有可以进行 Mock 测试(通过构造虚拟测试对 象以简化测试环境的方法)的伪后端，能支持前端的独立开发和测试。而后端 代码中除了功能实现外，还有着详细的测试用例，以保证 API 的可用性，降低 集成风险。

> 前端技术方案

- Vue.js：

Vue 是一套用于构建用户界面的渐进式框架。与其它大型框架不同的是，Vue 被设计为可以自底向上逐层应用。Vue 的核心库只关注视图层，不仅易于上手，还便于与第三方库或既有项目整合。另一方面，当与现代化的工具链以及各种支持类库结合使用时，Vue 也完全能够为复杂的单页应用提供驱动。

图

- Axios：

由于采用前后端分离的架构，因此使用 Axios 这一个基于 promise 的 HTTP 库，可以用在浏览器和 node.js 中。

- Ant Design Vue：

采用 Ant Design UI 组件库，ui 组件丰富齐全，能满足项目中大部分需求，开发效率高。

图

- Ckeditor4-vue：

功能强大，编辑能力强。

### 3.2 实习的收获与体会

#### 3.2.1 收获

首先，感谢我们团队的小伙伴们，有良好的团队规范、工作习惯，也有着非常融洽的工作氛围。在这个团队里，我主要的收获就是参与并熟悉了规范的公司项目迭代开发流程，并能较好的参与项目、融入团队。可以扮演好自己的角色，让我感觉十分荣幸。

其次，在个人提升方面，就是获得了前端技术在实际业务中的运用能力。在这之前，对于各项前端技术的学习基本上都是脱离实际应用的，曾经的项目经历只能称为一些个人 demo。在实际项目中高强度运用，编写真实的业务逻辑代码，大大巩固了这些知识的记忆，增加了处理实际问题的经验。同时，也学习到了作为一个标准项目，项目结构的合理性，组件封装的优缺点，一些刚性需求（例如调主题色）的实现方法与代价等，收获颇丰。

#### 3.2.2 体会

首次踏入职场，既有兴奋，也有紧张，好在一切顺利，基本能够完成学习任务与工作上的需求。

实际工作体验与校内的学习还是有着很大的不同，实际业务项目比起作业/比赛项目，需要考虑的方面也多出很多。一个项目不能无脑去做，要有完整的预期和规划；做一个简单的功能也不能只考虑实现，要考虑投入、性能优化等等其他东西……同时，这段时间以来，除上述之外，我自己的脑中还积累了很多方面的内容，需要一段时间沉淀整理。

团队里的小伙伴们技术高超，工作气氛融洽，也给了我很好的工作体验。

### 3.3 问题与探讨

#### 3.3.1 本项目 - 知识库

依我浅薄之见，知识库项目还存在一些各方面的问题。

首先，由于时间紧、人员变动、前端人力不足的问题，导致项目初期没有完全规划好。虽然启动阶段对技术栈有所调研，但是没有产出完整的方案文档，也没有制定好项目目录结构、命名规范、组件封装方式等。由于是团队协作开发的方式，每个人的代码风格等也都有所不同，所以这些都导致了开发过程中到处缝缝补补，进度有所影响。

其次，后端 API 接口开发速度快，但是文档更新不及时，我自己也没有做到与后端很好的沟通，在对接口数据的时候进度也比较缓慢。（在我原有印象中，采用前后端分离的开发方式的话，应该是先一起讨论制定好 API 文档、数据字段格式，再分头进行开发的？）现在的情况是前端需要真实数据的话必须等待后端开发完成、更新 API 文档后，才能进行数据对接，这也一定程度上影响了进度。
